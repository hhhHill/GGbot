# Hybrid Sync/Async LLM Execution Design

## 背景

当前项目已经具备：

- Web Chat 入口
- Agent 主循环
- Spring AI 聊天模型调用
- 文档 / PPT / 总结类工具

但在“请求执行方式”和“失败处理”上仍然是单一同步模型：

- 用户发送请求后直接等待当前 HTTP 请求返回
- 长耗时任务没有异步调度能力
- 没有统一的任务状态跟踪
- 前端无法持续显示任务进度
- 失败重试、超时兜底、看门狗治理还没有统一落位

项目目标不是做传统 UI 驱动的后端，而是逐步演进成 AI-native 系统。因此：

- 短请求要继续保留快速同步体验
- 长请求需要升级成任务化执行
- 用户应始终获得可理解的反馈

## 目标

本次设计目标是建立一套“同步优先，异步升级”的执行架构：

1. 普通短请求继续同步返回
2. 可能耗时的模型 / 工具请求升级为异步任务
3. 异步任务提交后立即返回唯一 `jobId`
4. Worker 后台执行任务并持续写入状态
5. 前端通过轮询展示进度
6. 任务卡死、模型失败、执行异常时有统一的超时、重试、兜底策略
7. 用户在失败后可直接点击按钮重传请求

## 非目标

本次设计不包含以下内容：

- 不实现 WebSocket / SSE 流式推送
- 不实现多用户并发任务调度优化
- 不引入外部消息队列中间件
- 不实现复杂优先级调度
- 不改造 Planner / Reflector 为 LLM 驱动

初版优先保证结构正确、行为一致、前后端链路清晰。

## 方案选择

### 方案 A：所有请求统一异步

优点：

- 行为统一
- 后端实现路径单一

缺点：

- 普通短请求也要轮询
- 用户体验变差
- 与当前 Web Chat 的快速问答场景不匹配

### 方案 B：前端先调用预判接口，再决定同步或异步

优点：

- 责任边界明确

缺点：

- 多一次请求
- 协议复杂度增加
- 收益有限

### 方案 C：入口统一，同步优先，后端判定是否异步升级

优点：

- 兼容当前接口
- 用户体验自然
- 后端可集中治理长任务

缺点：

- 需要一层异步判定策略

### 结论

采用方案 C。

## 总体设计

### 核心原则

执行方式只是外层包装，业务主链路仍然复用当前 Agent 核心。

也就是说：

- 短请求：`WebAgentController -> AgentService.handle(...)`
- 长请求：`WebAgentController -> JobService.create(...) -> JobWorker -> AgentService.handle(...)`

同步和异步不应发展成两套不同的执行逻辑。

## 后端架构

### 1. `AsyncExecutionDecider`

职责：

- 判断当前请求是否应走异步执行

输入：

- `AgentRequest`
- 或者基于请求生成的 `Plan`

输出：

- `SYNC`
- `ASYNC`

初版策略采用规则式判断，不做复杂预测。

建议：

- 普通短 `CHAT` 默认同步
- 明确的导出、复杂生成、多步工具任务走异步
- 后续可按工具类型、意图类型、步骤数扩展

### 2. `JobService`

职责：

- 创建异步任务
- 查询任务
- 更新任务状态
- 更新进度
- 标记成功 / 失败 / 超时
- 支持用户重传时复制原始请求生成新任务

建议新增独立模型：

- `JobRecord`
- `JobStatus`
- `JobService`

不直接复用当前轻量 `TaskService` 承载全部异步信息，避免职责混乱。

### 3. `JobWorker`

职责：

- 消费异步任务
- 执行现有 Agent 主链路
- 在关键节点写入任务进度
- 捕获异常
- 管理任务级重试
- 最终写入成功结果或失败兜底

关键原则：

- 不重写 Agent 执行逻辑
- 只作为异步任务包装器复用 `AgentService.handle(...)`

### 4. `JobWatchdog`

职责：

- 周期性扫描进行中任务
- 判断任务是否长时间无推进
- 对卡死任务执行超时收敛

判断依据：

- `status in (QUEUED, RUNNING, RETRYING)`
- `now - lastProgressAt > threshold`

动作：

- 标记 `TIMEOUT`
- 记录错误日志
- 写入用户可见兜底提示

## 模型调用可靠性层

### 目标

用户要求统一为全部模型调用生效，因此可靠性策略不应散落在具体工具或调用点里，而要放在统一模型访问层。

### 统一能力

建议在 `SpringAiChatService` 之上或内部抽象出统一可靠调用层，具备：

1. 空回复判定
2. 异常捕获
3. 最多 3 次重试
4. 每次重试之间有延时
5. 单次调用 60 秒超时
6. 超时 / 多次失败后的保底文案
7. 备用规则引擎答复

### 失败处理分层

为了避免职责混乱，失败处理分成三层：

#### 模型调用级重试

归属：

- `ReliableChatService` 或扩展后的 `SpringAiChatService`

职责：

- 处理单次模型调用的不稳定性
- 例如超时、空回复、临时异常

#### 任务执行级重试

归属：

- `JobWorker`

职责：

- 当整个异步任务执行失败时决定是否自动重跑任务

#### 用户级重传

归属：

- `POST /api/agent/jobs/{jobId}/retry`

职责：

- 用户点击失败后的“重传请求”按钮
- 创建新任务而不是覆盖旧任务

## 任务状态模型

建议固定为以下状态：

- `QUEUED`
- `RUNNING`
- `RETRYING`
- `SUCCEEDED`
- `FAILED`
- `TIMEOUT`

前端视角分组：

- 进行中：`QUEUED / RUNNING / RETRYING`
- 成功结束：`SUCCEEDED`
- 失败结束：`FAILED / TIMEOUT`

## `JobRecord` 建议字段

- `jobId`
- `conversationId`
- `userId`
- `status`
- `progressMessage`
- `progressPercent`，初版可选
- `retryCount`
- `createdAt`
- `updatedAt`
- `lastProgressAt`
- `resultPayload`
- `errorCode`
- `errorMessage`
- `fallbackReply`
- `originalRequestPayload`

初版建议优先使用：

- `progressMessage`

而不是追求精确百分比。

示例进度文案：

- `已接收请求`
- `正在分析任务`
- `正在调用模型`
- `正在整理最终结果`
- `正在重试第 2 次`

## 接口设计

### 1. `POST /api/agent/chat`

统一入口，返回两种响应之一：

#### 同步短请求

- 直接返回现有 `WebChatResponse`

#### 异步长请求

返回类似：

- `accepted=true`
- `jobId`
- `status=QUEUED`

### 2. `GET /api/agent/jobs/{jobId}`

职责：

- 前端轮询获取任务状态和结果

返回内容建议包含：

- `jobId`
- `status`
- `progressMessage`
- `retryCount`
- `canRetry`
- `resultPayload`
- `fallbackReply`

### 3. `POST /api/agent/jobs/{jobId}/retry`

职责：

- 用户失败后点击“重传请求”

行为：

- 读取旧任务原始请求
- 创建新任务
- 返回新的 `jobId`

推荐：

- 新任务使用新的 `jobId`
- 保留旧任务历史不变

## 前端设计

### 1. 同步短请求

用户点击发送后：

- 禁用输入框
- 禁用发送按钮
- 发送按钮显示旋转圈圈
- `CHAT` 区域显示一行小字：`正在生成中...`

完成后：

- 恢复输入与按钮
- 移除小字提示
- 展示最终回复

失败后：

- 恢复输入与按钮
- 显示失败提示
- 提供“重传请求”按钮

### 2. 异步长请求

提交成功拿到 `jobId` 后：

- 前端进入轮询模式
- 聊天区域插入占位消息

示例：

- `任务已提交，正在处理中...`

轮询中：

- 用 `progressMessage` 更新该占位消息

成功时：

- 用 `resultPayload.replyText` 替换占位内容

失败时：

- 占位消息替换为失败提示
- 显示“重传请求”按钮

### 3. 输入限制

初版策略：

- 同一会话只允许一个活跃中的请求
- 同步请求执行中禁止继续提交
- 异步长任务轮询期间也禁止继续提交

原因：

- 当前系统尚未准备好多任务并发会话语义

后续如果需要，可再放开并发。

## 数据流

### 同步链路

1. 用户调用 `POST /api/agent/chat`
2. `AsyncExecutionDecider` 判断为同步
3. 直接调用 `AgentService.handle(...)`
4. 返回最终结果

### 异步链路

1. 用户调用 `POST /api/agent/chat`
2. `AsyncExecutionDecider` 判断为异步
3. 创建 `JobRecord`
4. 立即返回 `jobId`
5. `JobWorker` 后台执行 `AgentService.handle(...)`
6. 持续写入状态与进度
7. 前端轮询 `GET /api/agent/jobs/{jobId}`
8. 成功或失败时结束轮询并展示最终状态

## 错误处理与兜底

### 模型级失败

- 先做模型级重试
- 若仍失败，尝试备用规则答复
- 若规则不适用，返回通用保底文案

### 任务级失败

- `JobWorker` 根据策略标记 `RETRYING`
- 超过任务级重试上限后标记 `FAILED`

### 看门狗超时

- 超过阈值且无推进，标记 `TIMEOUT`
- 返回兜底提示

### 前端失败提示

用户应始终看到清晰的失败反馈，例如：

- `请求处理失败，请稍后再试。`

并可直接点击：

- `重传请求`

## 测试

至少覆盖以下测试：

### 后端

1. 短请求命中同步路径
2. 长请求命中异步路径并返回 `jobId`
3. `JobWorker` 能执行并写入成功状态
4. 模型空回复时触发兜底
5. 模型异常时触发重试与兜底
6. 看门狗能将长时间无推进任务标记为 `TIMEOUT`
7. 重传接口会创建新的任务

### 前端

1. 发送后按钮进入 loading 态
2. 同步请求显示“正在生成中...”并在结束后消失
3. 异步请求会轮询并更新进度
4. 失败时显示失败文案和重传按钮

## 风险

### 1. 同步 / 异步判定策略过早复杂化

初版应使用简单规则，不要做预测模型。

### 2. 任务状态过度设计

初版优先保证：

- 状态正确
- 进度文案清晰

不急于追求复杂指标。

### 3. 可靠调用层和任务层职责混淆

必须保持：

- 模型级失败由聊天服务层处理
- 任务级失败由 Worker 处理
- 用户级重传由接口层触发

## 实施建议

建议顺序：

1. 先定义 `JobStatus` / `JobRecord` / `JobService`
2. 接入 `AsyncExecutionDecider`
3. 新增异步任务接口与轮询接口
4. 实现 `JobWorker`
5. 接入模型可靠调用层
6. 实现看门狗
7. 改造前端交互状态
8. 完成回归测试

## 结论

本次方案的核心不是简单给前端加一个 loading，而是建立：

- 短请求同步
- 长请求任务化
- 模型统一容错
- 前端统一状态管理

的一整套执行架构。

这样既保留当前 Web Chat 的即时体验，也为后续 AI-native 的动作型能力预留了稳定基础。
