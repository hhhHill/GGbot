# Web 页面优化与 Prompt 集中管理设计

## 背景

当前项目已经具备一个可用的 Web 入口，但页面文案和视觉呈现仍带有明显的阶段性演示痕迹，例如标题、说明文案和默认会话 ID 中包含 `MVP` 表述。与此同时，后端的 `system prompt` 仍以内联字符串形式分散在业务类中，不利于后续维护、审阅和统一调整。

本次改动聚焦两个目标：

1. 优化现有 Web 前端页面的视觉表达和产品化文案，移除与 `MVP` 相关的展示文字。
2. 将长文本 `system prompt` 从业务类中抽离，统一放到 `prompts` 目录管理。

## 范围

### 包含内容

- 调整 `src/main/resources/static/index.html` 中的标题、页眉说明、默认提示文案与展示信息。
- 调整 `src/main/resources/static/app.css`，在不引入新前端框架的前提下改进页面层次、版式和视觉质感。
- 保持 `src/main/resources/static/app.js` 的接口交互逻辑不变，只修改必要的默认展示文案或默认值。
- 在 `src/main/resources/prompts/` 下新增集中式 prompt 资源文件。
- 新增一个后端 prompt 读取组件，统一加载 classpath 下的 prompt 文本。
- 将当前内联的长 `system prompt` 迁移为资源文件读取方式。

### 不包含内容

- 不修改现有前后端 HTTP 接口协议。
- 不引入前端构建工具、组件框架或样式库。
- 不把普通业务字符串、工具描述、测试数据、默认用户 ID、页面标签等全部迁入 `prompts/`。
- 不做与本次需求无关的后端架构重构。

## 方案概述

### 一、Web 页面优化

页面继续沿用当前静态资源结构：`index.html` + `app.css` + `app.js`。本次不改变交互行为，只优化信息表达与视觉层次。

主要调整方向如下：

- 页面标题与页眉去除 `MVP` 表述，改为更正式、更中性的产品命名。
- 主标题与副标题从“演示链路”语气切换为“工作台/控制台”语气，降低试验感。
- 右侧状态区和结果区保留原有信息，但通过更清晰的卡片层级和留白改善可读性。
- 示例操作按钮、消息气泡、输入区和状态卡片保留现有结构，但统一颜色、边框、阴影与节奏，减少“临时演示页”观感。
- 默认会话 ID 去除 `web-mvp-session` 命名，替换为不带阶段标签的中性默认值。

### 二、Prompt 集中管理

为避免长 `system prompt` 分散在业务类中，本次将新增集中式 prompt 资源目录：

- 路径：`src/main/resources/prompts/`
- 命名方式：按用途命名，例如 `summarize-system-prompt.txt`

后端新增一个轻量组件负责从 classpath 读取 prompt 文本。业务类不再直接维护多行字符串，而是按文件名获取内容，再传入现有聊天服务。

该组件应满足以下要求：

- 调用简单，适合在工具类或服务类中直接使用。
- 资源缺失时抛出明确异常，避免静默返回空字符串导致行为不可预测。
- 读取结果应做基础清理，例如保留主体文本、移除仅由资源格式带来的多余尾部空白。

## 详细设计

### 前端结构设计

#### `index.html`

- 将 `<title>`、页眉标签、主标题、副文案替换为正式产品表达。
- 保留聊天面板、结果面板、状态卡片和示例按钮结构，避免 JS 选择器失效。
- 保留会话 ID 和用户 ID 输入项，但默认值使用不含 `MVP` 的命名。
- 当前能力说明文案改成能力陈述，而不是阶段说明。

#### `app.css`

- 保留当前暖色系基础，但增强对比、留白和层级，使页面更像可长期使用的控制台而不是概念页。
- 优化 hero 区与内容区的比例关系，减少顶部“宣传区”占比，提升主工作区权重。
- 加强输入区、消息区、结果区的分区边界和滚动体验。
- 保持移动端断点与自适应布局，不牺牲当前可用性。

#### `app.js`

- 保持消息发送、轮询、重试、健康检查逻辑不变。
- 仅调整与页面默认值相关的字符串，例如默认会话 ID。
- 不变更接口路径、请求体字段和结果面板渲染逻辑。

### 后端 Prompt 读取设计

新增一个资源读取组件，例如 `PromptTemplateRepository` 或语义相近的命名，职责如下：

- 输入：prompt 资源文件名
- 输出：文件内容字符串
- 数据来源：`src/main/resources/prompts/`

使用方式示意：

1. 工具类通过依赖注入拿到 prompt 读取组件。
2. 工具类在初始化或调用时读取对应 prompt。
3. 工具类将读取到的 `system prompt` 传给现有 `ReliableChatService`。

### 迁移策略

本次只迁移“长文本 system prompt”。已识别到的首个明确目标是 `SummarizeTool` 中的内联提示词。其他短字符串不纳入此次迁移。

这意味着：

- `SummarizeTool` 的 `CHAT_SYSTEM_PROMPT` 常量会被删除。
- 对应内容迁移到 `src/main/resources/prompts/summarize-system-prompt.txt`。
- `SummarizeTool` 改为从集中组件读取 prompt 内容。

## 错误处理

- 如果 prompt 资源不存在，应抛出明确异常并携带文件名，便于快速定位配置问题。
- 不在读取失败时隐式改用空 prompt。
- 当 LLM 不可用时，仍保留 `SummarizeTool` 现有的模板回复分支，避免影响离线兜底行为。

## 测试与验证

至少覆盖以下验证点：

- Web 首页能正常加载，且页面中不再出现与 `MVP` 相关的展示文字。
- 默认会话 ID 已切换到新的中性值。
- `SummarizeTool` 在 LLM 可用时仍能通过集中管理的 prompt 发起聊天调用。
- `SummarizeTool` 在 LLM 不可用时仍走现有模板回复逻辑。
- prompt 资源缺失时会抛出明确异常，而不是静默失败。

## 风险与取舍

- 本次不引入更复杂的 prompt 管理体系，例如变量替换、多语言版本或配置中心同步，以控制范围。
- prompt 资源化后，运行时更依赖 classpath 资源完整性，因此需要用测试覆盖资源缺失场景。
- 页面视觉优化仍建立在纯静态 HTML/CSS 上，收益高但边界明确，不会演变成前端工程改造。

## 实施结果预期

- Web 页面从“演示页”观感提升为更稳定、正式的工作台观感。
- 仓库中的长文本 `system prompt` 有统一落点，便于后续迭代和审阅。
- 本次改动不会破坏已有聊天、异步任务轮询与重试链路。
