# Chat Sidebar And Routing Retrospective

## 1. 新建会话后看起来没有进入对话页

**问题**

- 首页发送第一条消息后，URL 已经变化，但右侧看起来仍像主页

**根因**

- 前端早期会话模型和正式持久化模型分裂
- 页面最初并没有完全围绕 `conversation` 作为单一真源工作

**修复**

- 前端彻底切到 `/api/conversations`
- 首发消息由后端创建 `conversation`
- SSE 首个进度事件返回 `conversationId`
- 前端拿到 `conversationId` 后切到 `/chat/{conversationId}`

**教训**

- 会话模型必须统一，不能让前端继续依赖旧的临时模型

---

## 2. 聊天记录没有真正持久化

**问题**

- 页面能聊天，但历史记录并不完全可靠

**根因**

- 项目中同时存在 `session` 和 `conversation` 两套会话体系
- 网页聊天最初主要还在走旧链路

**修复**

- 前端列表改读 `/api/conversations`
- 会话详情改读 `/api/conversations/{id}/messages`
- 首发消息不再先建 session

**教训**

- 前端、后端、数据库必须共享同一个会话真源

---

## 3. 会话页刷新出现 Whitelabel Error Page 500

**问题**

- 在 `/chat/{id}` 页面直接刷新会报 500

**根因**

- `WebAppController` 用了相对转发：

```java
return "forward:index.html";
```

- 在深层路由下刷新时会转发失败

**修复**

- 改成绝对转发：

```java
return "forward:/index.html";
```

**教训**

- SPA 前端路由必须由服务端稳定兜底，否则前端静态资源调试会被污染

---

## 4. 左侧三点按钮看不见、点不到

**问题**

- 用户看到的只是标题过长后的省略号，不是菜单按钮
- 按钮也无法稳定点击

**根因**

- 按钮和标题区域叠放，视觉上和标题省略号混淆
- 刷新 500 让页面一度无法稳定加载最新静态资源

**修复**

- 把会话项改成两列布局：
  - 左边是标题
  - 右边是独立按钮位
- 三点按钮只在 hover 时显示，但位置独立存在

**教训**

- 交互按钮不要叠在文本区域上，应该有独立点击区域

---

## 5. 重命名和删除对话

**问题**

- 用户需要真实持久化的重命名和删除，不是前端假操作

**根因**

- 之前没有对应的后端接口和数据库删除链路

**修复**

- 新增：
  - `PATCH /api/conversations/{conversationId}/title`
  - `DELETE /api/conversations/{conversationId}`
- 删除时同步删除该会话消息
- 删除当前会话后，前端立即跳回 `/`

**教训**

- 涉及会话管理的 UI 操作必须直连真实持久化接口

---

## 6. 普通聊天缺少最近 5 轮上下文

**问题**

- 模型只能看到当前输入，无法理解“他 / 她 / 这个 / 那个”这类上下文指代

**根因**

- `SummarizeTool` 之前直接把当前 prompt 传给 `chatService`
- 没有拼接最近对话历史

**修复**

- 新增 `MemoryManager`
- 新增 `ContextAwareChatService`
- 统一构建：

```text
=== 最近对话 ===
USER: ...
ASSISTANT: ...

=== 当前问题 ===
USER: ...
```

- 最多只带最近 5 轮，即最近 10 条消息

**教训**

- 多轮聊天一定要显式控制上下文窗口，不能只传当前一句话

---

## 总结

这次问题最终归结为三类：

1. 会话真源不统一
2. SPA 深链刷新兜底不正确
3. 侧边栏按钮布局不合理

真正解决问题的关键不是继续在旧状态上打补丁，而是：

- 会话统一到 `conversation`
- 刷新统一转发到 `/index.html`
- 菜单按钮独立成明确的交互区

这样之后，聊天跳转、持久化、刷新和侧边栏操作都稳定了。
